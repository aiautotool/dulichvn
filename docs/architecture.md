# Architecture

This document describes the runtime architecture of Vinago+ as it is actually wired today. It is intentionally implementation-first: every box maps to a real file in the repo.

## High-Level Diagram

```
                   ┌──────────────────────────────────────────────────────┐

                   │           │                  │                        │

              ┌────────────────────────┴──────────┐   ┌───────────┴────────┐
```

## Process / Module Boundaries

The codebase is small enough to map directly:

| Concern | Module / File | Notes |
| --- | --- | --- |
| App shell, navigation, screens, data | `App.tsx` (5,172 lines) | Single-file Expo app. All React state, all content catalogs, all analytics calls live here. |
| Bootstrap | `index.ts` | `registerRootComponent(App)`. |
| Cloudflare Worker (API) | `src/worker.ts` | One route: `POST /api/itinerary-email`. Sends through Resend. |
| Static site hosting config | `wrangler.jsonc` | Binds `./dist` as `ASSETS`, SPA fallback, runs worker first on `/api/*`. |
| Web export | `package.json` → `npm run export:web` (`npx expo export --platform web`) | Produces `dist/`. |
| Cloudflare deploy (OAuth) | `package.json` → `npm run deploy:cloudflare` (`npx wrangler deploy --config wrangler.jsonc`) | Needs scoped `CLOUDFLARE_API_TOKEN`. |
| Cloudflare deploy (global key) | `scripts/deploy-worker.sh` → `scripts/deploy-cloudflare-global.sh` | Resolves account/zone, creates/updates the proxied DNS record, then runs Wrangler with the global key. |

## Request Flows

### 1. Web SPA boot

1. Browser hits `https://vinago.aiautotool.com/` (or `/anything-else`).
2. Wrangler routes the request to `env.ASSETS.fetch(request)` with `not_found_handling: "single-page-application"`. Unknown paths return `index.html`.
3. The Metro-bundled JS boots, calls `registerRootComponent(App)`, then mounts the React tree.
4. On first mount, `App.tsx` calls `initializeGoogleAnalytics()`. If `EXPO_PUBLIC_GA_MEASUREMENT_ID` is set and starts with `G-`, it injects the gtag script with `send_page_view: false` and queues `app_opened`.

### 2. Native iOS / Android boot

1. Native shell starts `MainActivity` / `MainApplication` (generated by Expo prebuild, see `android/`).
2. Same `App.tsx` mounts.
3. `Platform.OS !== 'web'` short-circuits GA injection, so all events go through `enqueueAnalyticsEvent` (AsyncStorage) and the Google sign-in path uses `expo-auth-session` with the `vinagoplus:/oauthredirect` scheme.

### 3. Google sign-in

1. `Google.useIdTokenAuthRequest(...)` builds a request whose redirect URI is `AuthSession.makeRedirectUri({ scheme: 'vinagoplus', path: 'oauthredirect' })`.
2. `signInWithGoogle` calls `promptGoogleSignIn()`.
3. On success, `googleResponse` carries an `idToken` (or `accessToken`); the app decodes the JWT payload, stores the profile in `authSession`, persists it to AsyncStorage, and remembers the `id_token` for later calls to the email endpoint.
4. On web, the OAuth popup returns to the dev/prod URL + `/oauthredirect`; on native, it returns to `vinagoplus:/oauthredirect`. These URIs must be whitelisted in Google Cloud Console.

### 4. Itinerary email

1. User taps "Email confirmation" on the AI screen.
2. If `EXPO_PUBLIC_ITINERARY_EMAIL_ENDPOINT` is set, the app does:
   - `fetch(endpoint, { method: 'POST', headers: { Authorization: 'Bearer <googleIdToken>' }, body: JSON.stringify({ to, name, itinerary, profile }) })`
3. The Worker (`src/worker.ts`):
   - Validates method (`OPTIONS` for CORS preflight, `POST` for the real call, else 405).
   - Verifies `RESEND_API_KEY` is set (else 503).
   - Decodes the Google ID token payload (no signature verification beyond `email_verified` — see [Integrations → Itinerary Email](./integrations/itinerary-email.md) for caveats).
   - Asserts `payload.to` matches the token email.
   - POSTs to `https://api.resend.com/emails` with `ITINERARY_EMAIL_FROM` (default `Vinago+ <itinerary@aiautotool.com>`).
   - Returns `{ ok, provider, id }` to the client.
4. If the endpoint env var is **not** set, the app skips the Worker entirely and opens `expo-mail-composer` with the itinerary prefilled.

### 5. Analytics

- **Web, GA configured:** `sendGoogleAnalyticsEvent` pushes through `window.gtag`.
- **Web, GA not configured / native:** `enqueueAnalyticsEvent` writes to `vinago-plus-analytics-queue` in AsyncStorage (capped at the last 100 events).
- Every event carries the same context: `app_name`, `platform`, `session_id`, `locale`, `language`, `current_city`, `purpose`, `trip_days`, `ga_property_id`, `ga_stream_id`. See [Integrations → Google Analytics](./integrations/google-analytics.md) for the full event schema.

## Configuration / Data Files

| File | What it stores | When it is read |
| --- | --- | --- |
| `app.json` → `extra.googleAnalytics` | GA property / stream metadata | Build time, embedded in the bundle |
| `app.json` → `android.package`, `ios.bundleIdentifier` | `com.vinago.plus` | Native build |
| `app.json` → `scheme` | `vinagoplus` | OAuth deep link |
| `wrangler.jsonc` → `vars.ITINERARY_EMAIL_FROM` | Sender identity for Resend | Worker boot |
| `wrangler.jsonc` → `routes[0]` | `vinago.aiautotool.com` custom domain | Worker deploy |
| `wrangler.jsonc` → `assets.run_worker_first` | `["/api/*"]` | Edge routing decision |
| `.env` / shell | `EXPO_PUBLIC_*` | Build time, exposed to client code |
| Wrangler secret | `RESEND_API_KEY` | Worker boot, runtime only |
| `google-services.json` | Android Firebase config | Native Android build |

| --- | --- | --- |
| `vinago-plus-profile` | `UserProfile` | Cleared on "Open profile settings" |
| `vinago-plus-favorites` | `SavedItem[]` | Cleared only by uninstall or app reset |
| `vinago-plus-auth-session` | `AuthSessionState` | Cleared on sign-out |
| `vinago-plus-activity-history` | `ActivityHistoryEntry[]` | Cleared by "Clear history" button on the History tab |
| `vinago-plus-recent-searches` | `RecentSearch[]` | Cleared by "Clear all" button on the Search screen |
| `vinago-plus-settings` | `SettingsState` | Cleared only by uninstall or app reset |
| `vinago-plus-analytics-queue` | `AnalyticsPayload[]` | Last 100 entries; consumed by a real GA sink in the future |

## Type-checker quirk

The in-file `translations` object has 220 keys typed as `keyof typeof translations.en`. With TypeScript 6.0, type-checking this union needs a large JS stack. The project already configures this in `package.json`:

```json
"typecheck": "node --stack-size=20000 ./node_modules/typescript/bin/tsc --noEmit"
```

If you run `npx tsc` directly, it will crash with "Maximum call stack size exceeded". Always use `npm run typecheck`.

## Failure Modes & Fallbacks

| Failure | Behavior |
| --- | --- |
| `EXPO_PUBLIC_GA_MEASUREMENT_ID` missing or wrong format | `initializeGoogleAnalytics` warns and returns. All events queue to AsyncStorage. |
| Worker has no `RESEND_API_KEY` | Returns 503 `{ error: "Email provider is not configured" }`; app shows `email.failed`. |
| Worker can't verify the recipient email | Returns 403; app shows `email.failed`. |
| Google sign-in throws / returns error | `signInWithGoogle` catches, records activity, fires `google_sign_in_failed`. |
| `expo-mail-composer` not available | `MailComposer.isAvailableAsync()` returns `false`; app shows `email.unavailable`. |
| AsyncStorage read throws on boot | App falls back to `profile = null` (re-shows onboarding) and proceeds. |
| Native Google sign-in uses wrong Android client | `google-services.json` was copied from a different Firebase project; see [Integrations → Google Auth](./integrations/google-auth.md) for the production checklist. |
| "Open in Maps" tapped without a maps app installed | Falls back to opening `https://www.google.com/maps/search/?api=1&query=lat,lng` via `Linking.openURL`. |
